Django upgrade (1.4 -> 1.9) in a 100+ app project

Romain Garrigues (@rgarrigues)

Context @ iwoca

  • August 2015: end of life for Django 1.4 LTS
  • Monolithic Django project:
    • 100+ custom apps
    • 200K lines of code
  • Continuous deployment (~10 releases/day)
  • 20 full-time developers

Problem to solve

  • Massive upgrade (1.4 to 1.8)
  • Upgrade current code while 1.4 code still produced

Solutions

  • Separate branch, then final merge
  • Update the project continuously
    • TODO: Create a schema to illustrate?

Process

  • One version a time (1.5, 1.6, 1.7, 1.8, 1.9)
  • Read Django release notes
  • Create env with new Django version
  • Run tests
  • Fix each issue

One fix at a time

  • Compatibility 1.4 <--> 1.X
  • Deprecated rules
  • Documentation

Compatibility

Examples

                
from django.db import transaction

if django.VERSION < (1, 6):
    atomic_compat_transaction = transaction.commit_on_success
else:
    atomic_compat_transaction = transaction.atomic

Examples

                
if django.VERSION < (1, 8):
    ALL_FIELDS = None
else:
    ALL_FIELDS = '__all__'


class YourForm(forms.ModelForm):

    class Meta:
        model = YourModel
        fields = ALL_FIELDS

  • Automatic script launched during test phase
  • Prevent regressions
  • Help people to fix it themself

Benefits

  • Each change has small impacts on production code
  • Final deployment is really trivial

Results

  • Only small problems, essentially custom django admin (non-covered)
  • One production AttributeError bug (investigate)
  • No stress during the whole upgrade (months)
